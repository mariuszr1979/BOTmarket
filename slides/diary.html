Step 3: Schema Store

Content-addressed schema registration. Same schema → same hash. Always. No labels, no categories, no taxonomy. Pure SHA-256.

How It Works

1. canonical_input = json.dumps( input_schema, sort_keys=True, separators=(',',':')) 2. canonical_output = json.dumps( output_schema, sort_keys=True, separators=(',',':')) 3. combined = input + "||" + output 4. capability_hash = sha256(combined).hexdigest() 5. INSERT OR IGNORE → idempotent

API

POST /v1/schemas/register Auth: X-API-Key header Request: { "input_schema": {"type": "text"}, "output_schema": {"type": "text"} } Response: { "capability_hash": "d4e5f6..." } ✓ 64-char hex SHA-256 ✓ Deterministic (key order irrelevant) ✓ Idempotent (10× same → 1 row)
PS#6 Discovery by Example — content-addressed, not label-addressed

Step 5: Match Engine

Buyer sends capability_hash → engine returns best seller. DNS model: query → answer. CU debited, escrow created, active_calls incremented. All atomic.

POST /v1/match

Auth: X-API-Key (buyer) Request: { "capability_hash": "d4e5f6...", "max_price_cu": 25, // optional "max_latency_us": 5000 // optional } Response (matched): { "trade_id": "uuid...", "seller_pubkey": "uuid...", "price_cu": 20.0, "status": "matched" } Response (no match): {"status": "no_match"} Response (can't afford): {"status": "insufficient_cu"}

Algorithm

1. Look up seller_tables[hash] → sorted by (price ASC, latency ASC) 2. For each seller: a. Skip if price > max_price_cu b. Skip if latency > max_latency_us c. Skip if at capacity d. MATCH → break 3. Verify buyer CU ≥ price → insufficient_cu if not 4. Atomic: - Debit buyer cu_balance - Create trade (status=matched) - Create escrow (status=held) - Increment active_calls - Record event
PS#4 Match, Don't Trade — no order book, no resting orders

Step 6: Trade Execution

Buyer sends input → exchange proxies to seller → measures latency → returns output. CU stays in escrow. Settlement is Step 7.

POST /v1/trades/{id}/execute

Auth: X-API-Key (must be buyer) Request: { "input": "Article text..." } Response: { "output": "executed:Article text...", "latency_us": 145, "status": "executed" } Errors: 404 — trade not found 403 — not the buyer 400 — not in matched status

Execution Flow

1. Look up trade → verify matched 2. Verify caller is buyer 3. start_ns = time.time_ns() 4. Forward to seller (MVP: simulated) 5. end_ns = time.time_ns() 6. latency_us = (end - start) / 1000 7. Update trade: status=executed 8. Decrement seller.active_calls 9. Record event: trade_executed 10. Return output + latency
● Escrow stays held → Step 7
● Latency measured deterministically
● active_calls freed for next match
